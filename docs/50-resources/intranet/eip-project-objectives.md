# EIP Project Objectives

## Introduction

Welcome to the Technical Track of the Epitech Innovative Project (EIP) – where engineering excellence
meets innovation. This track is designed for students who are passionate about pushing technological
boundaries, solving complex technical challenges, and building solutions that stand out through their ar-
chitectural elegance, performance, and reliability.

The Technical Track transforms you from a coder into a technical architect. You'll dive deep into emerging
technologies, master the art of making informed technical decisions, and learn to build systems that are
not just functional, but scalable, maintainable, and secure. This journey is about developing the technical
leadership skills that distinguish great engineers from good developers.

Throughout this track, you'll embrace the full spectrum of technical excellence. You'll conduct rigorous
technology evaluation and integration, architect robust and well-documented systems, engage with the
open-source community, collaborate with technical experts, and optimize for performance at every level.
Each objective pushes you to think critically about technology choices and their long-term implications.

This guide outlines the comprehensive framework for your Technical Track journey. It details mandatory
objectives that will challenge your technical depth, as well as complementary goals you'll select with your
mentor to align with your project's specific technical ambitions. Clear deliverables and evaluation criteria
ensure you understand what technical excellence means in practice.

While pursuing these technical objectives, remember that you'll continue developing your project's func-
tionalityandcollaborativeaspectsinparallel. Theworkloadwillfluctuatethroughouttheyear,withcertain
periods demanding deeper technical focus. Strong time management and regular communication with
your mentor and the EIP academic coordinator will be crucial. They're here to challenge your architec-
tural decisions, validate your technical choices, and help you navigate the demanding but rewarding path
of technical innovation.

## Mandatory Objectives to Achieve

### 1. Evaluating and Integrating New Technologies (Technology Watch)

#### Active Technology Watch
3 Set up a regular watchon topics linkedtothe project: languages, frameworks, protocols, tools, stan-
dards, architectures, etc.
3 Use varied, credible sources: technical blogs, scientific publications, official documentation, GitHub
repos, newsletters, Discord, X, etc.
3 Identify emerging trends/solutions that are useful for the project (not merely curiosity).
#### Documentation & Analysis
3 Write comparative briefs or analysis notes on the explored tech: advantages, drawbacks, learning
curve, project fit.
3 Produce one or more technical benchmarks between approaches/solutions (performance, scalabil-
ity, maintainability, ecosystem, etc.).
3 Present a reasoned decision on choices made (or rejected).
#### Concrete Application
3 Carry out 1–2 concrete experiments (POC, mini‑project, tested integration in the repo).
3 Integrate at least one new technology into the project with working proof and justification.
3 Update the project (docs, architecture, code) to reflect adopted evolutions.
#### Sharing & Openness
3 Participate actively in tech communities: exchanges on Discord, StackOverflow, GitHub, specialized
forums, etc.
3 Document your advances (README, changelog, wiki, GitHub issues), and share learnings within the
team.
#### Expected Deliverables (Non-exhaustive)
3 Structured watch notes or technical briefs.
3 Captures/exports of bookmarks, RSS feeds or newsletters followed.
3 Comparative sheets or benchmark tables.
3 POCs/mini‑projects testing technologies (separate repos or dedicated branches).
3 Changelog/README detailing the integration performed.
3 Documented rationale for final (or rejected) choices.
3 Community contributions (links/captures to GitHub, forums, Discord, etc.).
3 Impact synthesis on the project (performance, security, quality, dev time, ...).
#### What Is Not Sufficient
3 Reading a few articles with no testing or application.
3 Integrating a tech with no documentation or critical analysis.
3 Choosing a solution only because it’s trendy.
3 Reusing a tech already mastered by the team with no new exploration.
3 Producing nothing concrete from the watch.
### 2. Structure, Document, and Harden the Project’s Technical Architecture

#### Clear, Justified Architecture
3 Present the project’s architecture: components, services, dependencies, data flows, deployment,
versioning, etc.
3 Justify structural choices: selected technologies, code organization, databases, APIs, frameworks,
etc.
#### Complete Technical Documentation
3 Provide a structured README including:
- project purpose,
- install/deploy,
- folder/module structure,
- key commands,
- environment variables,
- technical prerequisites.
3 Create at least one advanced document: architecture diagram, technical use case, business logic,
technical roadmap, etc.

#### Quality, Reliability & Security
3 Enforce code standards (linting, conventions, naming, typing).
3 Run quality analysis tools (SonarQube, ESLint, Pylint, etc.).
3 Provide unit or validation tests on at least one critical module.
3 Implement initial error handling, logging, and minimal security (auth, sanitization, backups, ...).

#### Expected Deliverables (Non-exhaustive)
3 Complete README.
3 Architecture/deployment diagram.
3 Changelog or technical plan.
3 Example config files (dev/prod).
3 Code‑quality reports (ESLint, Sonar, etc.).
3 Screens/results of unit/stress/validation tests.
3 Technical wiki/Notion or annex documentation.
3 Install/deploy scripts (Dockerfile, bash script, etc.).

#### What Is Not Sufficient
3 A project with no README or an empty one.
3 A monolithic, non‑modular code structure.
3 No error‑handling or configuration management.
3 An unjustified stack (“because we know React”).
3 No documentation beyond the code itself.

## Complementary Objectives


### 1. Collaborate with Technical Experts

#### Identify & Approach Experts
3 Clearly define the technical need or topic to challenge: architecture choice, framework, security,
performance, scalability, AI, etc.
3 Proactively search for relevant experts: GitHub contributors, LinkedIn engineers, community leads,
teachers, CTOs, alumni, etc.
3 Professional outreach: clear message, precise question, appropriate channel.
Structured Collaboration
3 Run a documented exchange: meeting, written feedback, Discord/GitHub thread, code review, etc.
3 Optionally organize a review session with an external expert (pair programming, code review, light
audit).
3 Implement concrete actions from the exchange: structure adjustment, technical plan change, new
resources, tool adoption, process improvement.
#### Capitalize on the Exchange
3 Write a summary/memo: discussed points, advice received, decisions taken.
3 Add the collaboration to the roadmap or documentation.
3 Valorize when possible: testimonial, mention, quote, public thanks.
#### Expected Deliverables (Non-exhaustive)
3 Expert sheet: name, profile, link, role.
3 Outreach message (copy/capture/link).
3 Proof of exchange (meeting, Slack, message, GitHub comment...).
3 Memo or post‑collaboration analysis note.
3 Capture/list of changes implemented after the exchange.
3 Expert testimonial (optional but valuable).
#### What Is Not Sufficient
3 Sending a vague message (“can you look at our project?”) without a clear ask.
3 Citing a verbal/informal chat without trace or application.
3 Repeating only what the EIP mentor said without seeking an external view.
3 No notes taken, no change made after the exchange.
3 Contacting only close network (classmates, former teammates...).
### 4. Measure, Test, and Optimize Technical Performance

#### Define Key Technical Metrics
3 Choose 2–3 performance indicators relevant to the project (latency, response time, memory usage,
load time, bundle size, CPU, API calls, etc.).
3 Integrate these metrics into the dev cycle (dashboard, logs, third‑party tools, tests, etc.).
Set Up Automated/Manual Tests
3 Run load tests, stress tests, and/or simulation scenarios.
3 Run resilience tests: error handling, interruptions, unexpected behavior.
3 Run comparative efficiency tests: optimized vs. initial version.
#### Implement Optimizations
3 Analyze bottlenecks and friction points.
3 Modify one or more technical bricks (queries, processing, cache, code, ...).
3 Justify trade‑offs (quality vs. performance vs. cost vs. time).
#### Expected Deliverables (Non-exhaustive)
3 Chosen indicators + tracking table.
3 Before/after test results (logs, charts, captures).
3 Scripts/tools used for tests (e.g., JMeter.jmx, Docker, etc.).
3 Monitoring/benchmark dashboards (screenshots).
3 Analysis notes + applied optimization plan.
3 Technical rationale for improvements or encountered limits.
#### What Is Not Sufficient
3 Claiming “it runs fast on our machines”.
3 Blind optimization with no metrics or tests.
3 Relying only on intuition/habit.
3 Only reducing image/static file size without measured impact.
3 Failing to document obtained results.
